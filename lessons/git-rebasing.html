How Git Rebasing Works

Rewriting history sounds scary. It isn't — once you see that a commit is just a snapshot and a branch is just a sticky note. Let's build the whole idea from scratch.

01The 30-second version

Rebasing means giving your work a new starting point. You pick up the commits you made on your branch, set them aside, jump forward to the latest code, and re-apply your commits on top — one at a time — as if you'd started from there all along.

The result is a clean, straight line of history: no tangled merge commits, just your work sitting neatly on top of everyone else's. That's the entire idea. Everything below is just why it works and how to do it without hurting yourself.

02Three ideas you need first

You cannot understand rebasing until three small ideas are crystal clear. Most confusion about Git evaporates the moment these click. Take a minute on each.

① A commit is a snapshot with a parent

A commit is a photograph of your entire project at one moment, plus a pointer to the commit that came right before it — its parent. Because every commit points back to its parent, commits form a chain stretching all the way to the very first one.

Here's the part that makes rebasing make sense: a commit's ID (that a1b2c3 hash) is calculated from its contents and its parent. Change either one and you get a completely different commit with a different ID. Remember this — it's the secret of the whole article.

② A branch is just a movable label

This surprises everyone: a branch is not a box that holds commits. A branch is a tiny, movable sticky note that points at one commit — the latest on that line of work. Make a new commit, and the sticky note slides forward to it. That's all a branch is.

③ HEAD is "you are here"

HEAD is a pointer to where you currently are — normally it points at a branch (and therefore at that branch's commit). It's how Git knows which sticky note to slide forward when you commit.

🔑
Hold onto this

A commit's identity depends on its parent. So if you ever give a commit a new parent, you don't move the old commit — you create a brand-new commit with the same changes but a different ID. That single fact is rebasing.

03The problem rebase solves

You branch off main to build a feature. You make a couple of commits. Meanwhile, your teammates merge their own work into main, so main moves ahead. Now your branch is hanging off an old commit — your work is "based on" yesterday's code.

You'd like your feature to sit on top of today's main: partly to pull in the latest changes, and partly so that when you merge back, the history is clean instead of tangled.

04Two ways to catch up

There are two ways to bring your feature up to date with main. They reach a similar place but tell the story very differently. Flip between them:

The trade: beautifully linear, easy-to-read history. But you have rewritten your commits — the originals are replaced by new ones with new IDs. (Hold that thought; it matters for the golden rule.)

The trade: completely truthful and non-destructive — but the history becomes a web. Do this many times across many branches and the graph turns into spaghetti.

🧭
One sentence to remember the difference

Merge says "here is exactly what happened."  Rebase says "let me retell the story as if it had happened cleanly."

05How rebase actually works

The name gives it away: re-base = give your commits a new base. When you run git rebase main while standing on feature, Git performs a precise little dance:

1. Find where the two branches diverged — the most recent commit they share. In our picture that's B.
2. Take every commit on feature after that point — E, then F — and save them aside as a list of changes (think: patches).
3. Move feature's starting point to the tip of main — commit D.
4. Re-apply the saved changes on top of D, one at a time: D + E's changes → E′, then E′ + F's changes → F′.
5. Slide the feature label to the final new commit, F′.

Same changes, new commits

E′ does the same thing E did, but it is not E — it has a different parent and so a different ID. The originals (E, F) are now orphaned and quietly cleaned up later. This is why people say rebasing "rewrites history": it literally builds new history to replace the old.

06The commands

The everyday rebase is two lines. Stand on the branch you want to move, then name the branch you want to land on top of:

git switch feature      # 1. be on the branch you want to move
git rebase main         # 2. replay feature's commits on top of main

That's it — your feature now sits on the latest main. The common next step is to update main itself to include the finished feature. Because feature is now directly ahead of main, this is a clean fast-forward (no merge commit needed):

git switch main
git merge feature        # fast-forward: main just slides up to feature
✅
A handy shortcut

You can rebase a branch without switching to it first by naming both: git rebase main feature means "replay feature onto main." And to pull the latest main and rebase your current branch onto it in one move: git pull --rebase.

07Interactive rebase: editing history on purpose

So far rebase has just moved commits. But its superpower is letting you edit them as they replay — reorder, combine, rename, or delete. This is how you turn a messy stream of wip, oops, fix typo commits into one clean, reviewable story before anyone sees it.

Add the -i (interactive) flag and tell Git how far back to go:

git rebase -i HEAD~3       # edit the last 3 commits

Git opens a "to-do list" in your editor — your commits, oldest at the top, each with the verb pick in front. You edit the verbs to say what should happen to each commit:

reworda1b2c3Add login formfix the message
squashd4e5f6oops, fix typo in labelfold into the commit above
drop7a8b9cdebug console.logdelete it entirely
pick0d1e2fAdd password validationkeep as-is

The verbs you'll actually use:

VerbWhat it does
pickKeep the commit exactly as it is. (The default.)
rewordKeep the changes, but let you rewrite the commit message.
squashMerge this commit into the one above it, combining both messages.
fixupLike squash, but throw away this commit's message — perfect for "oops" commits.
editPause the rebase here so you can amend the commit's contents.
dropRemove the commit entirely. (Deleting the line does the same thing.)

You can also reorder commits just by moving their lines up or down. Save the file, close it, and Git replays everything according to your instructions. Three scrappy commits become one polished one.

🧼
The classic use

Right before opening a pull request, run git rebase -i to squash your "work-in-progress" commits into a couple of meaningful ones with clear messages. Reviewers see a clean story instead of your thinking-out-loud.

08The golden rule of rebasing

⚠️
Never rebase commits that other people already have.

In practice: don't rebase anything you've pushed to a shared branch (like main) that teammates may have pulled or built on top of.

Here's why, and it follows directly from everything above. Rebasing replaces commits with new ones that have new IDs. If you rewrite commits that someone else already has, your history and their history now disagree about what "the same commit" even is. When they next pull, Git sees two divergent versions of the past and you get duplicated commits, ugly conflicts, and a very confused team.

🛟
The safe zone

Rebase freely on your own local, unshared branches — that's exactly what they're for. The classic safe workflow: rebase your private feature branch onto the latest main right before opening or updating your pull request. Private history? Rewrite away. Shared history? Hands off.

09Force-pushing (safely)

Here's a wrinkle: once you rebase a branch you've already pushed, your local history no longer matches the remote's. A normal git push gets rejected — Git is protecting you from accidentally erasing remote commits. To publish your rewritten branch, you have to overwrite the remote copy. Do it with the safety catch on:

git push --force-with-lease

--force-with-lease only overwrites the remote branch if nobody else has pushed to it since you last looked. If someone has, it refuses — saving you from stomping on a teammate's work. Always prefer it over the blunt --force, which overwrites unconditionally.

⚠️
Force-push only your own branches

This is the golden rule in action. Force-pushing your personal feature/PR branch is normal and fine. Force-pushing a shared branch like main is how you ruin a Friday afternoon for the whole team.

10When things go wrong

Conflicts during a rebase

Because rebase re-applies your commits one at a time, a conflict can surface at each step (when a replayed change clashes with the new base). Git pauses, tells you which files conflict, and waits. You resolve it the same way you'd resolve any merge conflict, then keep going:

# edit the conflicted files to fix them, then:
git add <files>
git rebase --continue     # resume with the next commit

git rebase --skip         # skip the current commit (rarely needed)
git rebase --abort        # bail out and return to exactly how things were

It can feel like more work than a merge, but it's the same total effort spread into smaller, clearer pieces — and --abort always gives you a clean escape hatch back to the start.

The undo button: reflog

This is the safety net that makes rebasing low-risk: Git keeps a private journal of every place HEAD has been, called the reflog. Even after a rebase "rewrites" history, the old commits aren't truly gone yet — they're still there, just unlabeled. The reflog lets you find them and jump back:

git reflog                  # list everywhere HEAD has been, newest first
git reset --hard HEAD@{5}    # travel back to how things were 5 moves ago
🧯
Rebase with confidence

Because the reflog remembers the "before" state, almost any rebase can be undone. Knowing your --abort and reflog escape hatches is what lets you experiment freely instead of fearing every rebase.

11Merge vs rebase: which, when?

Neither is "better" — they're tools for different jobs. The deciding question is almost always: is this history private to me, or shared with others?

RebaseMerge
Resulting historyStraight, linear lineBranches with merge commits
Rewrites commits?Yes (new IDs)No (preserves originals)
Records true timeline?No — it's an idealized retellingYes — exactly what happened
Safe on shared branches?NoYes
Best for…Tidying your own branch before sharing; keeping main linearCombining shared branches; preserving real history
🧭
Rule of thumb

Rebase to tidy what's private. Merge to record what's public. A very common professional workflow combines both: rebase your feature branch onto the latest main to keep it clean and current, then merge the finished feature into main.

12Quick reference

Everyday rebase

Move your branch on top of the latest main.

git switch feature git rebase main

Clean up commits

Squash / reword / reorder the last N commits.

git rebase -i HEAD~N

Pull + rebase

Get latest and replay your work on top.

git pull --rebase

Publish a rewritten branch

Safely overwrite your own remote branch.

git push --force-with-lease

Mid-rebase controls

When a conflict pauses the rebase.

git rebase --continue git rebase --abort

Undo a rebase

Find the "before" state and jump back.

git reflog git reset --hard HEAD@{n}

The five things to remember

  • 🏷️A branch is just a movable label, and a commit is a snapshot plus a pointer to its parent.
  • 🔁Rebase = replay your commits onto a new base. Same changes, brand-new commit IDs.
  • ✨It trades the true history for a clean, linear one — great for readability.
  • ⚠️Golden rule: never rebase commits others already have. Rewrite private history only.
  • 🧯git reflog is your undo button — which is exactly why you can rebase fearlessly.
