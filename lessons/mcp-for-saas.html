01The model in one frame

MCP is a capability-negotiating RPC (JSON-RPC 2.0) between an AI host and external servers. The host runs one client per server; each client holds a 1:1 session. The value isn't any single tool — it's the standardized shape: any MCP client can talk to any MCP server, so integrations stop being bespoke.

02Primitives & the control model

The non-surface point: MCP's design is really a control model — each primitive answers "who decides this runs?" That's what makes it safe(ish) to give a model real capabilities.

PrimitiveExposed byControlled byIs
ToolsServerModelActions the model may call (tools/call)
ResourcesServerApp / userRead-only context the host attaches (files, records)
PromptsServerUserPre-built templates / workflows the user invokes
SamplingClientHostServer asks the host's LLM to complete — no own API key
ElicitationClientUserServer asks the user for input mid-run (human-in-the-loop)
RootsClientAppBoundaries — which paths/URLs a server may touch
🔁
The under-appreciated one

Sampling reverses the arrow: the server borrows the host's model. It enables "agentic" servers without their own keys — and is a real security surface, because a server can now drive inference.

03How an integration actually runs

The lifecycle is short: initialize (negotiate capabilities & version) → discover (tools/list, cacheable per ttlMs) → invoke (tools/call) → results back to the model. On the wire it's plain JSON-RPC:

tools/call — request & result

// host → server
{ "jsonrpc":"2.0", "id":7, "method":"tools/call",
  "params":{ "name":"get_invoice", "arguments":{ "id":"inv_88" } } }

// server → host  (content is returned to the MODEL as context)
{ "jsonrpc":"2.0", "id":7,
  "result":{ "content":[{ "type":"text", "text":"{ amount: 4200, status: 'due' }" }] } }

Note the last line: tool output flows straight into the model's context. Treat it as untrusted input — that's the root of MCP's security story (§07).

04Auth — the B2B-critical part

MCP deliberately doesn't invent auth — that's why it's enterprise-viable. A remote server is an OAuth 2.1 Resource Server. It advertises its authorization server via a .well-known/oauth-protected-resource document; the client gets a token from your IdP (Okta, Entra, Auth0) and calls the server with it.

  • Audience-bound tokens (RFC 8707 Resource Indicators) — a token minted for server A can't be replayed against server B.
  • Dynamic client registration so clients you didn't pre-provision can still onboard.
  • Per-tenant, narrowly-scoped tokens — one customer's agent can never reach another's data.
🔑
The takeaway

MCP rides OAuth/OIDC, so you reuse your existing IdP and authorization model. Scope tightly and bind audience — broad tokens are the #1 enterprise MCP footgun.

05Composing servers — how you "stack" them

Base MCP has no server-to-server calls: it's 1 client : 1 server, and the host fans out. To combine many servers — and to make that safe in production — you put a gateway in front.

Other composition moves: a proxy server (a server that is itself a client, wrapping another) and private registries that expose only org-approved servers.

06The two-sided B2B play

For a SaaS, MCP is two distinct moves — and the second is the strategic one:

The strategic read

Shipping an MCP server is the new public API — except the "developer" integrating you is an autonomous agent. It's a distribution channel and a disintermediation risk at once. MCP Apps let your server return interactive UI, clawing some of that surface back.

07Security failure modes (MCP barely specifies security)

The protocol standardized auth only recently and still leaves most security to you. The notable classes:

  • Tool poisoning — malicious instructions hidden in a tool's description; the model reads descriptions as guidance, so what it executes ≠ what you think the tool does.
  • Rug pulls — a server changes its tool definitions after you approved them. Safe on day 1, exfiltrating keys on day 7. Trust-on-first-use is not enough.
  • Confused deputy / cross-server shadowing — one server's tool description manipulates the agent into misusing another connected server's powerful tools.
  • Injection via tool results — untrusted data returned by a tool hijacks the agent (the lethal trifecta: private data + untrusted content + exfiltration).
  • Token theft & over-broad scopes — the classic OAuth failure, amplified because an agent wields the token autonomously.
🛡️
Mitigations that matter

Pin & diff tool definitions (catch rug pulls); vet/sign servers; narrow, audience-bound, per-tenant tokens; a gateway with allow-lists + policy + audit; human approval for high-impact tools; and treat every tool output as untrusted. See the OWASP MCP cheat sheet.

08Limits, pros & cons

✓ Pros

  • Write once, work everywhere — one server, every MCP client
  • Standard auth (OAuth/OIDC — reuse your IdP)
  • Dynamic discovery — tools appear without redeploys
  • Decouples tools from the model & the app
  • De-facto standard — Anthropic, OpenAI, Google, Microsoft, AWS
  • Turns your SaaS into an agent-callable surface

✕ Cons / limits

  • Security is largely your job (poisoning, rug pulls, injection)
  • Tool bloat — too many tools degrade model selection
  • It's plumbing, not orchestration — no agent logic, no planning
  • Versioning & discovery still maturing; spec moves fast
  • Stateful sessions vs scale; extra latency per hop
  • UI disintermediation risk once you expose

09MCP vs the neighbors

  • vs function calling / OpenAPI plugins: those are per-app and ad-hoc. MCP standardizes the shape + discovery + transport + auth, so an integration is reusable across every client.
  • vs A2A (Agent2Agent): complementary, not rival. MCP = agent ↔ tools/data (vertical); A2A = agent ↔ agent (horizontal, delegation between peers). A serious system uses both.
  • MCP Apps / OpenAI Apps SDK: built on MCP, adding interactive UI returned by tools (see Generative UI).
🏛️
For a regulated SaaS (e.g. financial advisory)

Front everything with a gateway (auth · allow-lists · audit), pin tool defs, scope tokens per-tenant, treat tool output as untrusted, log every call for recordkeeping, and gate money/advice behind human approval. MCP delivers the integration; your compliance spine governs it.

The comprehensive-but-short version

  • MCP = JSON-RPC capability negotiation; host runs 1 client per server; transports are stdio & Streamable HTTP.
  • The primitives encode a control model (model/user/app decides) — Tools, Resources, Prompts, Sampling, Elicitation, Roots.
  • Auth is OAuth 2.1, server-as-resource-server — reuse your IdP; scope & audience-bind tokens.
  • Compose many servers with a gateway (auth · RBAC · audit · routing) — base MCP has no server-to-server.
  • For B2B: consume for speed, expose for distribution — exposing is the strategic, and risky, move.
  • It barely specifies security: poisoning, rug pulls, injection are yours to defend. Plumbing, not orchestration.
