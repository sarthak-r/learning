Sealing a Filled Artifact

A generated form is ephemeral. The instant your client submits, you must freeze what they sent into an immutable record and prove who did what, when. Here's exactly what to capture — and how to make it tamper-evident.

01Two jobs — don't conflate them

A "snapshot" and an "audit log" solve different problems. You need both, in separate stores.

SnapshotAudit log
CapturesState — what was submittedEvents — what happened around it
ShapeOne immutable record (per version)Append-only stream of entries
Answers"What is the record?""How did it come to be? Was it tampered with?"
MutabilityWrite-once; edits = new versionAppend-only; never edit or delete

02What a snapshot must capture

Because the form was generated and personalized, you can't reconstruct it later by re-running the prompt — generation isn't deterministic. So freeze everything needed to reproduce the exact submission:

  • The answers — the data, validated against the schema.
  • The schema/contract version it was validated against.
  • The exact form presented — the rendered artifact (HTML or its spec) the client actually saw. Store it; don't regenerate.
  • Context — filler identity, tenant, locale, and the originating question/intent.
  • Integrity — a content hash (SHA-256) over all of the above; the hash becomes the snapshot's ID.
🔑
Freeze the presented form — never regenerate it

The record must be the exact thing the human saw and acted on. A regenerated form is a different artifact and won't hold up in a dispute or audit.

snapshot record (illustrative)

{
  "snapshot_id": "sha256:c71b9a…",        // = hash of this record's content
  "submission_of": "questionnaire/acme-kyc",
  "form_version": "fv_2f8c… (frozen)",    // schema + exact rendered artifact
  "answers": { "legal_name": "Acme Ltd", "turnover": 4200000 },
  "filled_by": { "id": "client_88", "email": "cfo@acme.com" },
  "tenant": "org_17", "locale": "en-GB",
  "sealed_at": "2026-06-04T09:12:03Z",    // server time
  "timestamp_token": "rfc3161:MIAGCSq…"   // proves it existed at T
}

03Make it tamper-evident

Immutability is a ladder — climb only as high as the data's stakes require:

TierAdd thisBuys you
MinimumContent-address: hash the snapshot; the hash is its IDAny change is detectable (new hash ≠ old)
StrongWORM storage (S3 Object Lock / append-only) + RFC 3161 trusted timestampCan't be edited/deleted; can't be back-dated
LegalDigital / e-signature (eIDAS · ESIGN · UETA), capturing signer identity + intentNon-repudiation — admissible as a signed record

A trusted timestamp is the cheap, high-leverage one: a third-party authority cryptographically attests the hash existed at time T, and nobody — including you — can backdate it. Open authorities (e.g. sigstore) make it a single call.

04The audit log — append-only, hash-chained

A separate, append-only store. Every meaningful event becomes one entry; entries are never edited. Log the whole lifecycle:

generated → shared → opened → field changed (optional) → submitted → sealed → viewed/exported/amended

Each entry records actor (id + role), action, target (snapshot id + version), trusted timestamp, and source (IP / device / session). Tamper-evidence comes from chaining: each entry embeds the hash of the previous one, so removing or altering any entry breaks the chain.

🔧
Two upgrades when it scales

Use HMAC-SHA256 (a keyed hash) so an attacker can't recompute valid hashes; switch to a Merkle tree when linear chain-verification gets slow (it gives logarithmic proofs).

05Seal server-side, on submit

The artifact is just a skin. It POSTs structured data to your backend — the spine — and the server does the sealing. The client never computes its own integrity; a hash or timestamp sent from the browser proves nothing.

💾
If you lean on a platform's artifact storage

Claude/ChatGPT artifact persistence is convenient, but it is not your system of record. Export and re-seal the submission into your own WORM + audit store; that's the copy you can govern, retain, and defend.

06Updates = new versions

"Living document" doesn't mean mutable. When the client or your user changes something later:

  • Write a new snapshot version — the previous one stays frozen, forever.
  • The new version links to its parent (a version chain), so you have the full lineage.
  • Append an audit event with who, when, and the diff.

Result: you can render the record as of any point in time and show exactly how it evolved — without ever overwriting history.

07Data model at a glance

StoreHoldsRule
form_versionsSchema + the exact frozen rendered artifactImmutable, versioned
snapshotsA submission: answers + context + hash, → a form_versionWrite-once; edits → new version
audit_eventsLifecycle events, each → a snapshot + versionAppend-only, hash-chained

Keep audit_events in a store with different, stricter write access than your app data — ideally one where deletes are blocked and recoverable.

08Pitfalls

  • Regenerating to "reconstruct." Freeze the presented form; never re-run the prompt to recreate it.
  • Trusting the client. Seal server-side; client-supplied hashes/timestamps are meaningless.
  • Mixing audit into app data. Separate store, append-only, restricted writes.
  • Dropping the schema version. Store it, or old submissions become uninterpretable as the contract evolves.
  • Client clocks. Use server / trusted time only.
  • Chaining but never checking. A chain only helps if you verify it on a schedule and anchor externally.
  • Forgetting PII duty. Encrypt at rest, log access, support retention & legal-hold.

Takeaways

  • Snapshot = state, audit log = events. Both, in separate stores.
  • Freeze the exact presented form — generated UI can't be reconstructed deterministically.
  • Content-hash → WORM → trusted timestamp is the high-leverage immutability stack; sign if it's a legal record.
  • Hash-chain the audit log, anchor it externally, and verify on a schedule.
  • Seal server-side; updates create new versions, never overwrite.
