Generative UI — Cut to the Signal

The term is buried under hype and "top 10 tools" listicles. Strip those away and there's a real, specific idea with a real production pattern. Here's the meat: what it is, what the sharpest people argue, and what's actionable now vs ambitious later.

01Definition — minus the hype

Generative UI = the interface itself — not just the text — is assembled by a model at runtime, from your data and the user's context. The model decides which UI to show and how to configure it, per user, per turn.

Most of the noise comes from conflating three different things. Separate them and the hype collapses into something concrete:

What people call "generative UI"Actually isWhen
Figma AI, Galileo, design copilotsAI design — produces mockups for humansDesign time
v0, bolt, LovableAI codegen — produces code a human shipsBuild time
Generative UI (proper)The running app renders UI the model chose for this user/turnRuntime

All three are useful. Only the third is the new idea. This page is about that one.

03The principle that makes it work

Why most "generative UI" demos fall apart in production: they let the model emit arbitrary code — inconsistent, inaccessible, untestable "AI slop." The fix is the whole game:

🎯
Constrain the model to choose & configure, never to code

Express intent in natural language, but compile it into a schema-validated composition over a fixed inventory of vetted components. The model selects a component and fills its props; your code renders it. You keep consistency, accessibility, brand, and testability — and still get adaptivity.

Concretely, the model's output is data, not markup — a spec your renderer maps to real components:

declarative UI spec — the model's output (illustrative)

{
  "component": "DataTable",        // chosen from your registry
  "props": {
    "columns": ["vendor", "spend", "status"],
    "rows": "$.invoices",          // bound to your data, not invented
    "sortBy": "spend"
  },
  "actions": [{ "label": "Approve", "intent": "approve_invoice" }]
}

Validate with JSON Schema / Zod; reject anything off-contract. This is the line between "adaptive product" and "unpredictable slop."

05Do now — the near-term, actionable bits

  • Swap text for widgets where input is structured. Dates → date picker, choices → buttons, results → table/chart/map. Wattenberger's affordances, applied. Highest signal-to-effort move.
  • Wire tool-calling to a component registry. The model returns which component + props; you render vetted ones. Off-the-shelf: Vercel AI SDK (streamUI), CopilotKit, thesys C1, or MCP Apps for cross-client widgets.
  • Constrain + validate. JSON Schema / Zod on every model-emitted spec; reject off-contract output. Start static, graduate to declarative.
  • Generate the arrangement, not the code. Personalize which components show and how they're ordered per user/role/task — over a fixed component vocabulary.
  • Keep fully-generated UI at build-time. Use v0/codegen with human review; don't ship raw model HTML to users.
⚡
The 1-line strategy

Your design system becomes the model's vocabulary. The model composes; it never invents pixels.

06The ambitious bet — where it's heading

  • Interface-for-one. The app assembles itself per user, task, and context — no two sessions identical.
  • Malleable software (Litt). Users reshape tools at the point of use; software becomes a material, not a shipped product.
  • Post-chat (Wattenberger, Lee). Rich, spatial, manipulable canvases co-driven by human + model replace the transcript.
  • Software 3.0 (Karpathy). Natural language is the programming layer; the durable asset shifts from screens → data + intent.
  • The long tail unlocks. Interfaces nobody could afford to hand-build — one-off, niche, per-customer — become generatable on demand.

07Honest caveats — what the same people flag

  • Non-determinism. Same prompt → different layouts; hard to test, review, govern. (The constraint pattern is the antidote.)
  • Discoverability / muscle memory. A UI that changes every time can't be learned. Stability is a feature; don't regenerate what users rely on.
  • Accessibility. Generated UI routinely skips a11y — it's a lagging afterthought in most of the hype. Bake it into the component library, not the prompt.
  • Latency, cost, trust. Every rendered turn is a model call; budget and cache accordingly.
  • It's a collaborator, not a replacement. Humans still set the vocabulary, guardrails, and taste. The model arranges within them.

Takeaways

  • Generative UI (proper) = runtime — the app renders model-chosen UI per user/turn. Not Figma-AI, not v0.
  • Keep one model: static → declarative → fully-generated. Production lives in the first two.
  • The whole game: model selects & configures vetted components, schema-validated — never emits raw code.
  • The thesis (Wattenberger, Lee, Litt, Karpathy): chat was a stopgap; the future is rich, governed, generated interfaces.
  • Now: widgets over text, design system as the model's vocabulary. Later: interface-for-one, malleable software.
