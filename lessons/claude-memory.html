01Two memory models

Claude doesn't "remember" — each API call starts cold. Memory is just a store of files the agent reads at the start of a task and writes as it learns. Anthropic ships this two ways, and they differ on the one question that matters: who holds the data.

Both are in beta (June 2026). The rest of this is how each works — and why a regulated B2B SaaS usually wants the first.

02The memory tool, on the wire

The client-side memory tool (memory_20250818) is just a tool Claude can call, scoped to a /memories directory. The twist: Anthropic doesn't execute it — you do. Claude emits the command; your app runs it against your storage and returns the result. Anthropic never sees the files, only the command log.

03Context editing — the other half

Memory only matters because the context window fills. Its partner feature, context editing (clear_tool_uses_20250919, same beta header context-management-2025-06-27), automatically clears the oldest tool results once input tokens cross a trigger (default 100k), keeping the few most recent.

♻️
The synergy is the mechanism for long-horizon agents

As the window approaches the limit, Claude is warned and writes the durable facts to memory before the transient tool results are cleared. Forget the noise, keep the signal. Anthropic reports ~84% token savings on long-running tasks with the two combined.

Knobs: trigger (when to clear), keep (recent tool pairs to retain, default 3), clear_at_least (for cache efficiency), exclude_tools. Clearing invalidates the prompt cache — clear_at_least blunts that.

04Memory in a Managed Agent

A Managed Agent is Anthropic's hosted agent runtime: Anthropic runs the loop, the sandbox, and tool execution; sessions are long-running and stateful. Here memory is not a client tool call — it's a first-class, server-side resource.

So "how memory works in a Managed Agent" = Anthropic stores it for you, versioned and audited, mounted as a filesystem the agent just reads and writes. No handler to implement — but the data lives on Anthropic's side.

07Limits & the real risk: memory poisoning

Persistence has a sharp edge: it turns a one-shot prompt injection into a durable compromise. If the agent processes untrusted input (a user message, a fetched page, a tool result) and gets injected, it can write attacker-controlled content into memory — which every future session then reads as trusted context.

☣️
Treat memory reads as untrusted

Mitigate: keep untrusted/reference material read-only; allow read-write only to agent-authored memory; isolate stores per tenant/session; sanitize before writing; validate paths (traversal); and review/audit what persists. Memory you can't trust is worse than no memory.

  • Beta, no GA (June 2026) — APIs and limits can shift.
  • Size — Managed: ~100 kB/file, 2,000/store; client-side: enforce your own caps & pruning.
  • Compliance — Managed Agents aren't ZDR/HIPAA-eligible yet; the client-side tool keeps data in your boundary.
  • Cost — memory tool: no separate charge; Managed Agents: ~$0.08/session-hour + model tokens.

08For a regulated SaaS

The pattern that fits a compliance-heavy B2B product:

  • Prefer the client-side tool — memory stays in your infrastructure (residency, encryption, retention, ZDR), not Anthropic's.
  • Per-tenant isolated stores — one customer's memory can never leak into another's context.
  • Audit every mutation — log each memory write as a record (the same WORM + audit spine your filings use).
  • Read-only by default for anything derived from untrusted input; human review of what an agent is allowed to persist about a client.

The framing

Memory + context editing give you a durable agent; your compliance spine governs what it's allowed to remember. The capability is Anthropic's; the trust boundary is yours.
