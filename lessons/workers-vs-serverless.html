Workers, Queues & Serverless

How a task gets handed to a worker, why long jobs belong off the request path, and when a worker beats a serverless function like Lambda. One idea organizes all of it: the queue is the boundary between the request path and the background path.

01The one boundary

There's a single decision rule: is a human blocked, right now, on this finishing? Yes → keep it on the request path, under the timeout. No → hand it to a worker on the background path. The queue is that boundary — and crossing it has a hidden cost: the request path is effectively exactly-once, but the background path is at-least-once — the same job will eventually run twice.

Request pathBackground path
Who's waitingA human, synchronouslyNo one — async
Time budgetSub-second → hard ~30s timeoutMinutes to hours
Delivery semanticsEffectively exactly-onceAt-least-once (plan for retries)
Generating a 40-yr projection?No — you'll blow the timeoutYes — enqueue it

02Why heavy work in the request path kills the box

It's not that the server can't do the math — a 40-year Monte Carlo costs the same CPU-seconds anywhere. It's that the request path has hard short deadlines and fixed concurrency:

  • Deadlines you don't control: API Gateway ~29s, ALB 60s, Heroku router 30s, Gunicorn worker 30s. The projection outlives them and gets killed mid-flight.
  • Fixed worker pool: a handful of slow projection requests occupy every web worker, so fast requests (a login, a page load) queue behind them — head-of-line blocking. A spike then becomes a retry storm and the box falls over. The crash isn't compute; it's starvation.

Offloading to a queue fixes this by decoupling arrival rate from service rate — the burst inflates queue depth instead of concurrent pressure.

03How a task actually flows (nobody "assigns" it)

The mental model "assign a task to a worker" is subtly wrong: producers don't pick workers — workers pull tasks. Many workers compete on one queue; the broker hands each message to exactly one. The queue is the load-balancer.

the boundary, in code

# request path — does almost nothing, returns in ~30ms
@app.post("/plans/{id}/generate")
def generate(id):
    enqueue("generate_plan", plan_id=id)   # pass the ID, NOT the data
    return 202, { "job_id": id, "status_url": f"/jobs/{id}" }

# background path — minutes later, on a worker, MUST be idempotent
@worker.task(acks_late=True)
def generate_plan(plan_id):
    if already_done(plan_id): return     # at-least-once → may run twice
    save(plan_id, run_projection(plan_id))  # the 40-year Monte Carlo

05The reliability machinery (the real price of the background path)

You traded a blocked human for a delivery guarantee that's weaker than it looks. At-least-once is universal; exactly-once delivery is a myth — the ack itself can be lost, so the broker must redeliver. You don't get exactly-once; you engineer effectively-once:

  • Idempotency is mandatory. Every handler runs twice eventually — dedupe on a key (plan_id + hash(inputs)) or make the write naturally overwrite-by-key.
  • Visibility timeout > p99, or heartbeat. The gotcha: acks_late does not extend the redelivery clock — a 45-min job under a shorter window gets a second worker started mid-flight.
  • Retries with backoff + jitter; a DLQ for poison messages (after N receives); graceful drain on deploy so in-flight jobs finish.
  • Pass IDs, not payloads. A fat message is a stale snapshot by the time it runs, bloats the broker, and busts size caps (SQS 256 KB). Big blob → claim-check (put it in S3, enqueue the key).
  • Mind the dual-write race. Commit-then-enqueue can lose a job; enqueue-then-rollback creates a phantom. Fix with a transactional outbox.

06Concurrency & scaling

  • Match the model to the work: prefork processes for CPU-bound projection math (the Python GIL means threads won't parallelize it); async/event-loop for I/O fan-out (calling many APIs).
  • Autoscale on queue depth, not CPU. A worker mid-projection pins a core whether the backlog is 0 or 10,000 — CPU is the wrong signal. Scale replicas on depth/lag (KEDA), targeting 60–80% utilization. Use a job-style scaler so scale-down can't evict a running 30-min job.
  • The DB is the real ceiling. Connections = replicas × concurrency × pool size; autoscaling workers can take down Postgres faster than they help. Cap with a pooler (PgBouncer) + a max-replica guard.

07Serverless & the 15-minute wall

A serverless function (AWS Lambda) is the other way to run the background path: event-driven, per-invocation isolated, scale-to-zero compute. You don't run a pool — each event gets its own ephemeral micro-VM. The trade-offs are sharp and mostly fixed:

  • Hard 900-second (15-min) timeout, 128 MB–10 GB RAM, stateless. A multi-decade projection simply won't fit in one invocation.
  • Idle = $0, billed per millisecond — but at a premium, with no commit discounts (Fargate/EC2 get Savings Plans ~50% and Spot ~70%).
  • Cold starts are real but a red herring for async batch jobs (no user waiting). SnapStart/Provisioned Concurrency mitigate when latency matters.
🧱
2026: the wall is getting escape hatches

Lambda Durable Functions checkpoint/replay a single execution for up to a year; Managed Instances run Lambda on your EC2 for steady workloads; Cloud Run lifted request timeouts to 60 min. The 900s per-invocation wall hasn't moved — but you can now step around it.

09It's converging — stop thinking binary

The two ends are growing into each other: serverless gained duration and state (Durable Functions, Managed Instances, Cloud Run 60-min); workers gained serverless ergonomics (KEDA scale-to-zero, Fargate, managed Temporal). "Function vs worker" is now mostly a billing-model + duration choice over the same container runtime — both billing models live on both sides of the boundary.

🧩
The shape for a plan-generation pipeline

For a multi-step, must-finish job, reach for durable execution (Temporal / Step Functions / Lambda Durable Functions): an orchestrator decomposes the work, Lambda runs the light steps, and a Fargate task runs the heavy projection. One caveat — durable execution moves the idempotency burden to each step boundary; it doesn't lift it. Your activities still need dedupe keys.

The short version

  • The queue is the boundary: request path (fast, human waiting, exactly-once) vs background path (slow, retried, at-least-once).
  • Heavy work crashes the request path through starvation (timeouts + fixed pool), not raw compute.
  • Workers don't get assigned work — they pull it (competing consumers); the queue load-levels bursts.
  • At-least-once is universal → idempotency is mandatory; pass IDs not payloads; autoscale on queue depth, not CPU.
  • Serverless = idle-$0, instant scale, 15-min wall. Workers = sustained-util cheap, unbounded duration.
  • It's converging: a billing-model + duration choice over the same runtime; durable execution for must-finish pipelines.
